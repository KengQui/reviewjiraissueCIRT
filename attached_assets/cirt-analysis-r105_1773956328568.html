The right agent received the request but its tools or APIs don't support the action, or there's a logic bug preventing it from completing the task.
There's a structural reason these issues are hard to recover from. The ExitAgentTool only has two reason codes — OUT_OF_SCOPE and TASK_COMPLETE. There's no CAPABILITY_LIMIT signal. When an agent hits the edge of what it can do (e.g., Timesheet can't calculate hours for a specific pay rule), there's no way to send that back to the orchestrator — the agent just responds in plain text, and the orchestrator never finds out. Because no signal fires, no fallback or re-routing kicks in either. The user is left to figure out what to do next on their own. That's why Pattern 3 issues can't be softened at the orchestrator layer the way Pattern 1 and 2 issues can — only the agent's own wording can save the experience.

How to address agent-level issues

Agent bugs need two types of fixes: code fixes from each domain team, and behavior specs that define how each agent should communicate when things go wrong. Here's what applies now vs. what comes next.

R105 — What the orchestrator can do now
The orchestrator sits between the user and every agent. Even without fixing the agents, these spec rules already in place soften the impact.
Spec RuleWhat it doesHelps with
FB-02 When an agent fails, show a clear plain-language message instead of a raw error code or silence Pattern 3 & 4
GRD-07 Never fabricate data — if the agent can't get it, don't guess Pattern 1
FMT-06 Recap what was collected before submitting — gives user a chance to catch missing fields even if the agent didn't validate Pattern 2
GRD-11 Require explicit confirmation before any write action — safety net even if agent processed data incorrectly Pattern 1 & 2
R106+ — What needs to happen next
The orchestrator spec defines how Bryte behaves at the routing layer. Each agent also needs its own behavior spec — a short set of rules defining what it says when it can't complete a task, returns an error, or encounters a validation issue. Right now each domain team makes that up on their own.

3 · The Spec's Real Value
The spec is a quality standard, not a bug fix

Counting bugs fixed is the wrong metric. The spec's value shows up in four ways:

Prevention
Categories of issues that never get filed because the rules exist upfront. The 8 "exposing internals" issues would never have been raised if the spec was in place from day one.
Consistency
Every agent handles errors, confirmations, and fallbacks the same way. Without the spec, each domain team does it differently, and the user experience is inconsistent.
Speed of resolution
When CIRT files a bug, the spec gives a rule to point to. Instead of debating "how should Bryte handle this?", teams can reference a specific rule (e.g., "FB-02 says...") and implement it.
Eval criteria
The spec enables systematic testing. Instead of CIRT poking around hoping to find problems, teams can write evals directly from spec rules and catch violations before CIRT does.
4 · Gaps & Recommendations
Three actions to increase coverage
R105 (now) — Ship the orchestrator spec
The spec covers 87% of behavioral issues. Add a localization section (4 rules) to close the remaining gap and reach 100% behavioral coverage. Add a response validation principle so the orchestrator catches agent errors before they reach the user.
R106 — Agent-level behavior specs
Write short behavior specs (5–10 rules each) for the two highest-pain agents: Timesheet (6 issues) and Direct Deposit/Payroll (5 issues). These 2 specs would cover 11 of the 19 agent-level issues. Partner with domain teams to implement.
R107+ — Expand agent specs
Roll out behavior specs to remaining agents (Absence, Accruals, Schedule, I-9, Course). Use CIRT data and eval results to prioritize which agents need specs next.

6 · The Bottom Line
Summary for leadership

The orchestrator behavior spec covers 87% of behavioral issues found by CIRT. It directly addresses how Bryte talks, routes requests, handles errors, and protects against information leaks.

The remaining CIRT issues (58%) are agent-level code bugs — wrong calculations, missing validations, broken features. These require engineering fixes from each domain team, and are outside the spec's scope by design.

The spec's biggest value is not in patching existing bugs, but in preventing future ones and creating a shared quality standard that all teams build against. With one additional section (localization), the spec achieves full coverage of its domain.
